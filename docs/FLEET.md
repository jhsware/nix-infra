# Fleet Guide

Fleet mode manages standalone machines or groups of independent servers. Unlike cluster mode, fleet mode does not set up etcd, overlay networking, or a service mesh — it's a simpler approach for servers that don't need service discovery or load balancing.

## When to Use Fleet vs Cluster

**Use fleet mode when you need:**

- A single server or a set of independent machines
- Simple deployments where each machine runs its own services
- Machines that don't need to discover or communicate with each other through a service mesh
- A quick setup without the overhead of certificates, etcd, or overlay networking

**Use cluster mode when you need:**

- High availability with multiple control nodes
- Service discovery and automatic load balancing (etcd + confd + HAProxy)
- An encrypted overlay network between nodes (Flanneld + WireGuard)
- Dynamic service registration and health checking

## Prerequisites

Before getting started, you need:

- The `nix-infra` binary installed
- SSH and OpenSSL available on your workstation
- A provider configured — either a Hetzner Cloud API token (`HCLOUD_TOKEN`) or a `servers.yaml` file for self-hosted servers (see [Providers](./PROVIDERS.md))
- A `.env` file with your environment variables (or set them in your shell)

## Project Structure

A fleet project typically has this structure (based on the `nix-infra-machine` template):

```
my-fleet/
├── .env                    # Environment variables (SSH_KEY, SSH_EMAIL, etc.)
├── configuration.nix       # Base NixOS configuration template
├── flake.nix              # Nix flake for reproducible builds
├── servers.yaml           # Self-hosted server definitions (if applicable)
├── ssh/                   # SSH key pairs (generated by init)
├── secrets/               # Encrypted secrets
├── node_types/            # Node module definitions
│   ├── web-server.nix     # Example: web server configuration
│   └── db-server.nix      # Example: database server configuration
├── modules/               # Shared NixOS modules
├── app_modules/           # Application configurations
│   ├── my-app/
│   │   ├── default.nix    # App NixOS module
│   │   └── action.sh      # Action script for operational tasks
│   └── another-app/
│       └── default.nix
└── nodes/                 # Per-node configuration overrides
    ├── web-1/
    │   └── apps.nix       # Which app_modules this node runs
    └── db-1/
        └── apps.nix
```

## Getting Started

### Step 1: Initialize the Project

```bash
nix-infra init --no-cert-auth --batch
```

This creates SSH keys for node authentication. The `--no-cert-auth` flag skips certificate authority creation since fleet mode doesn't use etcd or TLS certificates.

Required environment variables (in `.env` or shell):

| Variable | Description |
|----------|-------------|
| `SSH_KEY` | Name for the SSH key pair |
| `SSH_EMAIL` | Email address for the SSH key |

### Step 2: Provision Nodes

```bash
nix-infra fleet provision \
  --node-names "web-1 web-2" \
  --nixos-version 24.11 \
  --machine-type cx22 \
  --location fsn1
```

This creates cloud servers (Hetzner Cloud), waits for them to be ready, and converts them to NixOS using nixos-infect. For self-hosted servers defined in `servers.yaml`, provisioning installs NixOS on existing machines.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `--node-names` | Yes | Space-separated list of node names to create |
| `--nixos-version` | Yes | NixOS version to install (e.g., `24.11`) |
| `--machine-type` | Yes* | Server type (e.g., `cx22` for Hetzner Cloud) |
| `--location` | Yes* | Data center location (e.g., `fsn1`, `nbg1`, `hel1`) |
| `--placement-group` | No | Place nodes in a placement group (Hetzner Cloud only) |
| `--mutation` | No | Use a nixos-infect mutation for non-standard OS (e.g., VMware) |

*Required for cloud providers; ignored for self-hosted servers.

### Step 3: Initialize Machines

```bash
nix-infra fleet init-machine \
  --target "web-1" \
  --node-module web-server \
  --nixos-version 24.11
```

This deploys the machine configuration and application modules, then rebuilds NixOS. It performs three steps internally: deploy the machine configuration (`deployMachine`), deploy application configurations (`deployAppsOnNode`), and trigger a `nixos-rebuild switch`.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `--target` | Yes | Space-separated list of target nodes |
| `--node-module` | Yes | Node type from `node_types/` directory |
| `--nixos-version` | Yes | NixOS version |

### Step 4: Deploy Applications

```bash
nix-infra fleet deploy-apps \
  --target "web-1 web-2"
```

Deploys or updates application configurations on target nodes. Each node's `nodes/<name>/apps.nix` determines which `app_modules/` are deployed.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `--target` | Yes | Space-separated list of target nodes |
| `--test-dir` | No | Alternative directory to read app configurations from (useful for testing) |
| `--rebuild` | No | Whether to run `nixos-rebuild` after deploying (default: `true`) |

## Day-2 Operations

### Update Machine Configuration

```bash
nix-infra fleet update \
  --target "web-1" \
  --node-module web-server \
  --nixos-version 24.11 \
  --rebuild
```

Updates the base machine configuration. Use this when you've changed a node module and want to apply it.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `--target` | Yes | Target nodes |
| `--node-module` | Yes | Node type from `node_types/` |
| `--nixos-version` | Yes | NixOS version |
| `--rebuild` | No | Run `nixos-rebuild` after update (default: `true`) |

### Destroy Machines

```bash
nix-infra fleet destroy --target "web-2"
```

Destroys cloud-provisioned machines. Only works with providers that support server destruction (e.g., Hetzner Cloud). For self-hosted servers, remove them from `servers.yaml` instead.

### Upgrade NixOS

```bash
nix-infra fleet upgrade-nixos \
  --target "web-1" \
  --nixos-version 25.05
```

Performs a NixOS version upgrade. For major version changes (e.g., 24.11 → 25.05), this updates the nix channel, deploys the new configuration, and rebuilds. For minor updates within the same channel, it updates packages and rebuilds.

### Rollback

```bash
nix-infra fleet rollback --target "web-1"
```

Rolls back to the previous NixOS configuration using `nixos-rebuild switch --rollback`. This is useful if a deployment caused issues.

### Garbage Collect

```bash
nix-infra fleet gc --target "web-1"
```

Runs `nix-collect-garbage -d` on target nodes to free disk space by removing old system generations. **Warning:** This clears rollback history — you won't be able to roll back to previous configurations after garbage collection.

### SSH Access

```bash
# Interactive shell
nix-infra fleet ssh --target "web-1"

# Run a command
nix-infra fleet cmd --target "web-1 web-2" -- uptime
```

`ssh` opens an interactive terminal session on a single node. `cmd` runs a command on one or more nodes and returns the output.

### Port Forwarding

```bash
nix-infra fleet port-forward \
  --target "web-1" \
  --local-port 8080 \
  --remote-port 80
```

Forwards a remote port to your local machine over SSH. Useful for accessing services that aren't exposed publicly.

### Run Action Scripts

```bash
nix-infra fleet action \
  --target "web-1" \
  --app-module my-app \
  --cmd setup \
  --env-vars "DB_HOST=10.0.0.5,DB_PORT=5432"
```

Executes an action script (`app_modules/<app-module>/action.sh`) on target nodes. Action scripts are used for operational tasks like database migrations, cache clearing, or service initialization.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `--target` | Yes | Target nodes |
| `--app-module` | Yes | App module containing the action script |
| `--cmd` | Yes | Command to pass to the action script |
| `--env-vars` | No | Comma-separated environment variables |
| `--save-as-secret` | No | Save the action output as an encrypted secret |

The `--save-as-secret` option is useful for actions that generate credentials or tokens — the output is captured and stored as an encrypted secret that can be referenced in configurations with `[%%secrets/<name>%%]`.

### Upload Files

```bash
nix-infra fleet upload \
  --target "web-1" \
  --file "./data/config.json" \
  --remote-path /root/uploads
```

Uploads files or directories to target nodes via SFTP. Supports uploading entire directory trees.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `--target` | Yes | Target nodes |
| `--file` | Yes | Space-separated local file or directory paths |
| `--remote-path` | No | Remote destination path (default: `/root/uploads`) |

## Key Differences from Cluster Mode

| Feature | Fleet Mode | Cluster Mode |
|---------|-----------|--------------|
| Overlay network | No | Yes (Flanneld + WireGuard) |
| Service discovery | No | Yes (etcd + confd) |
| Load balancing | No | Yes (HAProxy) |
| TLS certificates | No | Yes (PKI for etcd) |
| Certificate authority | Not needed | Required |
| Init command | `init --no-cert-auth` | `init` (creates CA) |
| Node addressing | Direct IP | Overlay IPs |
| Variable substitution | `[%%<node>.ipv4%%]` | `[%%<node>.overlayIp%%]` also available |
| Secrets deployment | Via SSH, no overlay | Via SSH over overlay network |

In fleet mode, the `overlayNetwork` flag is set to `false` internally, which means secrets deployment uses direct IP addresses rather than overlay IPs, and overlay-specific variable substitutions are not available.

## Global Options

These options are available on all fleet subcommands:

| Option | Description |
|--------|-------------|
| `--working-dir`, `-d` | Working directory (default: `.`) |
| `--ssh-key` | SSH key name (overrides `SSH_KEY` from `.env`) |
| `--env` | Path to `.env` file |
| `--debug` | Enable verbose debug logging |
| `--batch` | Run non-interactively (skip confirmation prompts) |

## MCP Integration

For AI-assisted fleet management, nix-infra provides an MCP server (`nix-infra-machine-mcp`) that exposes fleet operations as tools. This allows AI assistants to help with monitoring, diagnostics, and routine operations while respecting safety restrictions (see [Security — MCP Server Safety](./SECURITY.md#mcp-server-safety)).
